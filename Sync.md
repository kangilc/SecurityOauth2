  | 종류 | 설명 | 장점 | 단점 | 주요 사용 사례 |
  | :--- | :--- | :--- | :--- | :--- |
  | 실시간 (Real-time) | 원본 데이터가 변경되는 즉시, 거의 지연 없이 동기화됩니다. (보통 이벤트 기반 아키텍처, 메시지 큐 등을 사용) | - 데이터 정합성이 가장 높음<br>- 사용자가 항상 최신 데이터 확인 가능 | - 시스템 부하가 큼<br>- 구현이 복잡하고 비용이 높음 | - 은행 거래 시스템<br>- 실시간 재고 관리<br>- 항공권 예약 시스템 |
  | 준-실시간 (Near Real-time) | 수 초에서 수 분 사이의 매우 짧은 주기로 동기화됩니다. (마이크로 배치 방식) | - 실시간에 가까운 데이터 제공<br>- 시스템 부하를 어느 정도 제어 가능 | - 실시간보다는 약간의 지연 발생<br>- 구현이 비교적 복잡함 | - 모니터링 대시보드<br>- 실시간 로그 분석<br>- 위치 기반 서비스 |
  | 배치 (Batch) | 정해진 스케줄(시간, 일, 주, 월 등)에 따라 주기적으로 동기화됩니다. 가장 전통적이고 일반적인 방식입니다. | - 시스템 부하 예측 및 관리가 용이<br>- 대용량 데이터 처리에 효율적<br>- 구현 및 운영이 비교적 단순 | - 데이터 지연(Latency) 발생<br>- 실시간성이 중요한 업무에는 부적합 | - (매일) 야간 ETL 작업, 데이터 웨어하우스<br>- (매시간) 통계 리포트 생성<br>- (매월) 월말 정산 및 빌링 |
  | 수동 (Manual) | 사용자가 직접 '동기화' 버튼을 누르거나 특정 액션을 취했을 때만 동기화가 실행됩니다. | - 사용자가 원하는 시점에만 실행<br>- 불필요한 시스템 리소스 낭비 없음 | - 자동화 불가<br>-사용자가 잊으면 데이터 불일치 발생 | - 스마트폰 주소록 동기화<br>- 클라우드 스토리지 파일 동기화 |
